## Environment Setup

### 1. Copy environment template

```bash
cp .env.development.example .env
```

### 2. Review `.env` configuration

Required variables (defaults are safe for local/demo):

| Variable | Default | Notes |
|----------|---------|-------|
| `APP_ENV` | `development` | Set to `production` for public demo |
| `POSTGRES_PASSWORD` | `postgres` | **Change for production** |
| `REDIS_PASSWORD` | `redispassword` | **Change for production** |
| `MINIO_ACCESS_KEY` | `minioadmin` | **Change for production** |
| `MINIO_SECRET_KEY` | `minioadmin` | **Change for production** |
| `JWT_SECRET_KEY` | *(empty)* | **Must set to strong random string** |
| `OPENAI_API_KEY` | *(empty)* | Required for LLM features |
| `ANTHROPIC_API_KEY` | *(empty)* | Required for LLM features |

Optional variables (leave empty to disable integrations):

| Variable | Purpose |
|----------|---------|
| `NOTION_TOKEN` / `NOTION_DATABASE_ID` | Notion MCP integration |
| `ASANA_TOKEN` / `ASANA_WORKSPACE_ID` / `ASANA_PROJECT_ID` | Asana MCP integration |
| `GITHUB_TOKEN` / `GITHUB_OWNER` / `GITHUB_REPOSITORY` | GitHub MCP integration |
| `BROWSER_API_URL` / `BROWSER_API_KEY` | Browser MCP integration |

### 3. Verify no secrets in git

```bash
git ls-files | grep -E '\.env$|\.key$|\.pem$' || echo "No secrets tracked"
```

## Database Initialization

### 1. Start infrastructure services first

```bash
docker compose up -d postgres redis qdrant minio
```

Wait for health checks to pass:

```bash
docker compose ps   # All should show "healthy"
```

### 2. Run Alembic migrations

```bash
make init-db
# Or manually:
docker compose exec backend alembic upgrade head
```

Verify migration succeeded:

```bash
docker compose exec postgres psql -U postgres -d ai_intelligence_os \
  -c "SELECT schema_name FROM information_schema.schemata WHERE schema_name NOT IN ('information_schema', 'pg_*');"
```

Expected output includes `public` schema.

### 3. Seed demo data (optional)

```bash
docker compose exec backend python -m database.docker.seed_demo
```

See `database/docker/README.md` for details.

## Service Startup

### Full stack (recommended for demo)

```bash
make start
```

This will:
1. Copy `.env.development.example` → `.env` if missing
2. Build all containers
3. Start PostgreSQL, Redis, Qdrant, MinIO, Backend, Frontend
4. Wait for health checks

## Health Verification

### 1. Check all services are healthy

```bash
docker compose ps
```

All services should show `healthy` status. If any show `unhealthy`, check logs:

```bash
docker compose logs <service-name> --tail=50
```

### 2. Verify backend health endpoints

```bash
curl -s http://localhost:8000/api/health | python3 -m json.tool
curl -s http://localhost:8000/api/live
```

Expected: `/api/health` returns `{"status": "healthy", ...}`, `/api/live` returns `200`.

### 3. Verify metrics endpoint

```bash
curl -s http://localhost:8000/metrics | head -30
```

Expected: Prometheus text exposition format with metric definitions.

### 4. Verify frontend loads

Open http://localhost:3000 in a browser. The page should load without console errors.

### 5. Verify database connectivity

```bash
docker compose exec postgres pg_isready -U postgres
```

Expected: `accepting connections`.

## Known Limitations

### v0.1.0 Beta

1. **Authentication is minimal** — JWT auth exists but lacks user registration UI and password reset flow. Use `JWT_SECRET_KEY` for token generation.

2. **LLM providers require API keys** — No built-in model hosting. Set `OPENAI_API_KEY` and/or `ANTHROPIC_API_KEY` in `.env`. Fallback providers configured via `config/models.yaml`.

3. **MCP integrations are optional** — Notion, Asana, GitHub, and Browser integrations require individual API tokens. Leave empty to skip.

4. **No multi-tenancy** — Single organization per deployment. User isolation is at the application level only.

5. **Frontend observability is client-side only** — Metrics flush to backend `/metrics` POST endpoint; no separate frontend metrics collector.

6. **Tracing exports to console only** — OpenTelemetry spans export to stdout. No OTLP collector or distributed trace backend configured by default.

7. **No horizontal scaling** — Docker Compose runs single instances. Redis is used for queueing but not clustering.

8. **MinIO uses default credentials** — `minioadmin/minioadmin` for object storage. Change in production.

9. **Database schema is evolving** — `init.sql` contains placeholder table definitions. Full schema is managed through Alembic migrations.

10. **No CI/CD pipeline** — Manual deployment only. Docker Compose is the primary deployment target.

## Release Steps

1. Create release branch: `git checkout -b release/v0.1.0-beta`
2. Bump version: update `VERSION` file to `0.1.0`
3. Run full test suite: `make test`
4. Review all changes: `git diff master...release/v0.1.0-beta`
5. Merge to master: `git checkout master && git merge release/v0.1.0-beta`
6. Tag release: `git tag -a v0.1.0-beta -m "Release v0.1.0 Beta"`
7. Push: `git push origin master --tags`
